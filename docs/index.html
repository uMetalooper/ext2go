Introduction

ext2 (Second Extended Filesystem) was the dominant Linux filesystem through the 1990s and early 2000s. It was designed by Rémy Card and released in 1993, replacing the original ext filesystem. For over a decade it was the default on nearly every Linux distribution.

ext2 has no journaling — no write-ahead log, no crash recovery mechanism. That's exactly what makes it the ideal filesystem to learn from. There is no hidden runtime state, no background threads, no complex recovery paths. It is just bytes on disk, arranged in well-defined structures at well-defined offsets. Once you understand ext2, you understand the core of every filesystem that came after it. ext3 and ext4 are ext2 with features bolted on — the fundamentals are identical.

Why learn ext2?

No journaling means no complexity. Every concept — superblock, block groups, inodes, directory entries — is visible and static on disk. Reading an ext2 image is like reading a book: there is no magic state.

A filesystem image is just a file

An ext2 "disk" is a flat binary file. Every structure lives at a fixed byte offset. You can open it with any hex editor and find the superblock at byte 1024, the magic number 0x53 0xef at bytes 1080–1081, and so on.

On-Disk Layout

The disk is divided into fixed-size blocks (1KB, 2KB, or 4KB). Blocks are grouped into block groups. Each group is self-contained: it has its own bitmaps, inode table, and data blocks. The layout of the entire disk looks like this:

For a 4MB image with 1KB blocks, there is exactly one block group containing all 4096 blocks. Larger filesystems span multiple groups — the kernel tries to keep each file's inode and data blocks in the same group to minimize disk seeks.

The Superblock

The superblock is the root of everything. It always lives at byte offset 1024 on the disk and is exactly 1024 bytes long. The first thing any filesystem driver does is read the superblock and check the magic number. If the magic doesn't match, it's not ext2. 

Byte offset 1024:
┌──────────────────────────────────────────────────────────────┐
│  s_inodes_count   (4B)  │  s_blocks_count   (4B)  │ ...     │
│  s_free_blocks    (4B)  │  s_free_inodes    (4B)  │ ...     │
│  s_first_data_block (4B)│  s_log_block_size (4B)  │ ...     │
│  s_magic = 0xef53 (2B)  │  s_state    (2B)        │ ...     │
│  s_uuid           [16B] │  s_volume_name   [16B]  │ ...     │
└──────────────────────────────────────────────────────────────┘
s_magic = 0xef53

Always the second field checked (after reading). If this is wrong, abort — this is not an ext2 filesystem. Stored little-endian so on disk it appears as 0x53 0xef.

s_log_block_size

Block size = 1024 << s_log_block_size. So: 0 → 1KB, 1 → 2KB, 2 → 4KB. This single field controls almost all offset calculations in the driver.

s_first_data_block

Which block number contains the superblock. 1 for 1KB blocks (block 0 is boot), 0 for 2KB+ blocks (superblock fits inside block 0 after the boot sector). The Group Descriptor Table is always at s_first_data_block + 1.

Superblock backups

ext2 writes copies of the superblock into the first block of every block group (and sometimes only certain groups in the "sparse superblock" feature). fsck uses these to recover a corrupted primary superblock.

Block Groups

The disk is divided into equally-sized block groups. Each group is independent — it has its own accounting structures and a slice of the total inode and data space. The Group Descriptor Table (a 32-byte struct per group, located right after the superblock) tells the driver where to find the bitmaps and inode table for each group.

Block Bitmap

One bit per block in the group. 0 = free, 1 = allocated. The driver scans this to find free blocks when writing new data.

  • Always 1 block in size
  • Limits group size to BlockSize × 8 blocks
  • 1KB block → max 8192 blocks/group

Inode Bitmap

One bit per inode slot in the group. 0 = free inode, 1 = in use. Scanned when creating a new file or directory.

  • Always 1 block in size
  • Max inodes per group = BlockSize × 8
  • Set at filesystem creation time

Inode Table

A contiguous array of fixed-size inode structs (128 bytes each for rev0, configurable in rev1). Every file and directory maps to exactly one row in this table.

  • Inode #1 = bad blocks inode
  • Inode #2 = root directory (always)
  • Inodes #3–10 = reserved by ext2

Data Blocks

The remaining blocks in the group store the actual content of files and directories. The block bitmap tracks which of these are in use.

  • File data blocks
  • Directory entry blocks
  • Indirect pointer blocks

The group descriptor struct (bg_block_bitmap, bg_inode_bitmap, bg_inode_table) gives the absolute block number of each structure. To reach the inode table for group 0: read bg_inode_table × BlockSize bytes into the file.

Inodes

An inode is a 128-byte struct that represents a file, directory, symlink, or device. Inodes do not store filenames. Filenames live in directory entries. An inode only knows about its content and metadata.

Field Size Description

To locate inode number N on disk: 

group_index = (N - 1) / inodes_per_group
local_index = (N - 1) % inodes_per_group
offset = group[group_index].inode_table * block_size + local_index * inode_size

Inode numbers are 1-based. Root is always inode 2.

Block Pointer Tree

The i_block[15] array is a small tree that maps a file's logical byte range to physical disk blocks. The first 12 entries are direct pointers — they point straight to data blocks. The remaining 3 entries add levels of indirection to support larger files.

With 1KB blocks and 4-byte block numbers, each indirect block holds 1024 / 4 = 256 pointers. A file over 12KB triggers the single-indirect pointer; over 268KB triggers double-indirect. In practice, the vast majority of files fit entirely in the 12 direct blocks.

Short symlinks are stored inline

If a symlink target is ≤ 60 bytes, ext2 stores it directly inside the i_block array itself (interpreting the 60 bytes as a character string), avoiding an extra block allocation entirely.

i_blocks counts 512-byte sectors

i_blocks in the inode counts 512-byte units regardless of filesystem block size. A file using 3 data blocks of 1KB each has i_blocks = 6 (3 × 2 sectors). This field is only used by du and stat, not for reading the file.

Directory Entries & Path Resolution

A directory is just a file with a special i_mode type bit. Its data blocks contain a packed, variable-length linked list of directory entry records. Each record maps one filename to one inode number. 

struct ext2_dir_entry {
    uint32  inode;       // inode number this name points to (0 = deleted)
    uint16  rec_len;     // byte length of this record including name + padding
    uint8   name_len;    // actual byte length of the name string
    uint8   file_type;   // 1=regular, 2=dir, 3=char, 4=block, 5=fifo, 6=sock, 7=symlink
    char    name[];      // name bytes — NOT null terminated
};

Records are packed end-to-end within a block. rec_len is always rounded up to a 4-byte boundary. The last record in a block has a rec_len that extends to the end of the block, absorbing any slack space. Deleting an entry sets inode = 0 and merges its space into the previous record by extending that record's rec_len.

Path Resolution: /docs/notes.txt

  1. Start at inode #2 (root directory — always hardcoded). Read its data blocks as directory entries.
  2. Scan entries for name == "docs". Found: inode = 13.
  3. Load inode #13. It is a directory. Read its data blocks as directory entries.
  4. Scan entries for name == "notes.txt". Found: inode = 15.
  5. Load inode #15. It is a regular file. Read its block list and concatenate data blocks up to i_size bytes.

Every path traversal — no matter how deep — is exactly this loop: load inode, read directory entries, find matching name, follow inode number, repeat. There is no inode cache in ext2 itself; that's the kernel's job.

Filenames are not in the inode

The inode has no idea what it's named. Multiple directory entries in different directories can point to the same inode number. That's a hard link — and it's why mv within the same filesystem is instantaneous: it just adds a new directory entry and removes the old one. No data moves.

Hard Links vs. Inodes

Because filenames and inodes are separate, the same inode can be referenced by multiple directory entries — potentially in different directories. This is a hard link.

Hard Links

Multiple directory entries, same inode number. Each link increments i_links_count. Deleting one link decrements it.

  • Cannot cross filesystem boundaries
  • Cannot link directories (prevents cycles)
  • All links are equal — no "original"
  • File freed only when count reaches 0

Unlinking (deletion)

The unlink() system call removes a directory entry and decrements i_links_count. The inode and data blocks are freed only when the count reaches 0 and no process has the file open.

  • This is why open files survive rm
  • i_dtime records when count hit 0
  • Block bitmap bits cleared on free

Symbolic Links

A different inode type (i_mode bits = 0xA000). The "content" is a path string. Can cross filesystem boundaries. The target may not exist.

  • Short targets (≤60 chars) stored in i_block directly
  • Long targets stored in a data block
  • i_links_count is always 1

What ext3 and ext4 Add

ext2 became ext3 and ext4 through incremental additions. The core structures — superblock, block groups, inodes, directory entries — are preserved in all three. The magic number changes, and new feature flags in the superblock gate the new capabilities.

Feature ext2 ext3 ext4
Journaling None Journal inode Journal + checksums
Block addressing 32-bit 32-bit 48-bit
File size limit 16 GB (1KB blocks) 16 GB (1KB blocks) 16 TB
FS size limit 4 TB 4 TB 1 EB
Large files (data structure) Block pointer tree Block pointer tree Extents (B-tree)
Directory indexing Linear scan Linear + HTree option HTree (hash B-tree)
Nanosecond timestamps No No Yes
Online defrag No No Yes (e4defrag)
ext3 journaling

The journal is stored in a special inode (inode #8 by default). Before writing metadata, the kernel logs the intended change to the journal. After a crash, fsck replays the journal instead of scanning the whole disk — mount time goes from minutes to seconds.

ext4 extents

Instead of 15 block pointers, ext4 inodes store a B-tree of extents — each extent is a (start block, length) pair describing a contiguous run. A 1GB file stored contiguously needs only 1 extent instead of thousands of block pointers. Reduces fragmentation and speeds up large file I/O.

ext2go: A Working Implementation

To make these concepts concrete, I built ext2go — a read-only ext2 filesystem reader in Go, written from scratch in ~300 lines across 7 files. It uses encoding/binary to parse the raw on-disk structs with no dependencies.

Commands

  • ext2go <image> info Superblock metadata — UUID, block size, inode count, block group layout
  • ext2go <image> ls [path] List directory entries with permissions, timestamps, and sizes
  • ext2go <image> read <path> Print file contents to stdout by resolving the block pointer tree
  • ext2go <image> stat <path> Show raw inode fields: mode, timestamps, links, and the full block list

disk.go

Raw I/O wrapper over the image file. readAt(offset) and readBlock(n) are the only primitives everything else uses.

superblock.go

Parses the superblock struct at byte 1024. Validates the magic number. Computes block size and inode size from the raw fields.

group.go

Reads the Group Descriptor Table. Each 32-byte entry gives the block numbers of the bitmaps and inode table for one block group.

inode.go

Parses inode structs and resolves the block pointer tree — walking direct, single-indirect, double-indirect, and triple-indirect pointers to produce a flat block list.

dir.go

Reads directory entries from a directory inode's data blocks. Walks the packed rec_len-linked list and returns (inode, name) pairs.

fs.go

High-level API: mount(), path resolution by walking inodes, ls(), readFile(), and stat().

Key Takeaways

A filesystem is just a binary format. Every structure — superblock, bitmaps, inode table, directory entries — has a fixed location and layout on disk. There is no magic state you cannot inspect.

Inodes and names are separate. An inode represents a file's content and metadata. Directory entries map human-readable names to inode numbers. This separation enables hard links, atomic renames, and open-file survival across deletion.

The block pointer tree scales through indirection. 12 direct pointers cover small files efficiently. Three levels of indirect blocks extend coverage to 16+ GB without changing the inode size.

ext3 and ext4 are ext2 plus features. Journaling, extents, and larger limits are layered on the same core concepts. Understanding ext2 means understanding all three.
